The Problem

1
Tuning kills the server. Restarting vLLM inside the pod kills PID 1, which crashes the entire pod. The agent gets stuck in a recovery loop.
2
No baseline comparison. Without a stable reference, you can't tell if a change actually improved performance or if variance is fooling you.
3
Manual tuning doesn't scale. vLLM has 10+ tunable parameters with complex interactions. Testing combinations by hand is slow and error-prone.
# What used to happen: agent → kill PID 1 to restart vLLM pod → CrashLoopBackOff agent → retry... retry... retry... agent → "Max iterations reached" # No results. No comparison. # Just a dead pod and wasted API tokens.
3 / 7

Our Solution: Pod-per-Experiment

1
Baseline is read-only. The original pod is never modified. It serves as a stable reference for every comparison.
2
Fresh pod per experiment. Each tuning attempt creates a new pod with different vLLM args, benchmarks it, then deletes it.
3
Automated A/B testing. Every experiment is compared against the same baseline with regression detection (2% threshold).
4
Self-cleaning. Experiment pods are deleted after benchmarking. Cleanup runs on exit, even on crashes.
# New architecture: Baseline pod (read-only, never modified) • port-forward :8000 • run_benchmark → baseline.json • fetch_vllm_logs → baseline config Experiment pod (created per tuning attempt) • port-forward :8001 • run_benchmark → experiment.json • compare_benchmarks(baseline, experiment) • delete pod (cleanup)

Stopping Criteria

The agent keeps exploring until 10 consecutive experiments show no improvement (>2% threshold). No premature stopping.

Key Features

🧪

Isolated Experiments

Each tuning attempt runs in its own pod. No risk of corrupting the baseline. Failed experiments are simply deleted.

📊

Smart Benchmarking

GuideLLM benchmarks with configurable profiles (balanced, decode-heavy, prefill-heavy, long-context). Auto-detects chat vs. completion models.

🔍

Deep Log Parsing

120+ regex patterns extract vLLM config, memory usage, attention backend, CUDA graphs, compilation timing, and errors from server logs.

🛡

Regression Detection

Metric-aware comparisons: throughput (higher=better), latency (lower=better). 2% threshold with directionality-aware flagging.

🔬

Kernel Profiling

PyTorch trace analysis with CUDA kernel categorization (attention, MoE, GEMM, communication) mapped to vLLM source files.

💰

Cost Tracking

Full Claude API token accounting with cache hit rates and estimated cost. CPMT (Cost Per Million Tokens) for inference cost analysis.
