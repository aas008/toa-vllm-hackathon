The Problem

1
Tuning kills the server. Restarting vLLM inside the pod kills PID 1, which crashes the entire pod. The agent gets stuck in a recovery loop.
2
No baseline comparison. Without a stable reference, you can't tell if a change actually improved performance or if variance is fooling you.
3
Manual tuning doesn't scale. vLLM has 10+ tunable parameters with complex interactions. Testing combinations by hand is slow and error-prone.
# What used to happen: agent → kill PID 1 to restart vLLM pod → CrashLoopBackOff agent → retry... retry... retry... agent → "Max iterations reached" # No results. No comparison. # Just a dead pod and wasted API tokens.
3 / 10

Our Solution: Pod-per-Experiment

1
Baseline is read-only. The original pod is never modified. It serves as a stable reference for every comparison.
2
Fresh pod per experiment. Each tuning attempt creates a new pod with different vLLM args, benchmarks it, then deletes it.
3
Automated A/B testing. Every experiment is compared against the same baseline with regression detection (2% threshold).
4
Self-cleaning. Experiment pods are deleted after benchmarking. atexit cleanup runs even on crashes.
# New architecture: Baseline pod (read-only, never modified) • oc port-forward :8000 • run_benchmark → baseline.json • fetch_vllm_logs → baseline config Experiment pod (created per tuning attempt) • oc port-forward :8001, :8002, ... • run_benchmark → experiment.json • compare_benchmarks(baseline, experiment) • oc delete pod (cleanup) # Real pod names from our run: vllm-tune-1777144117 (chunked prefill) vllm-tune-1777144304 (gpu-mem 0.95) vllm-tune-1777144451 (gpu-mem + prefix)

Stopping Criteria

The agent keeps exploring until 10 consecutive experiments show no improvement (>2% threshold). No premature stopping. Default max: 100 iterations.

System Prompt Engineering

Structured Workflow in the Prompt

TUNING WORKFLOW (follow this order strictly): 1. Benchmark the BASELINE pod (already running): a. Call run_benchmark with profile="balanced" b. Call fetch_vllm_logs (baseline pod logs) c. Call read_benchmark_results → Save the baseline JSON path for comparison. 2. For EACH tuning experiment: a. Call create_vllm_pod with vllm_args → Returns pod_name and endpoint b. Call run_benchmark with experiment endpoint c. Call fetch_vllm_logs with pod_name d. Call read_benchmark_results e. Call compare_benchmarks(baseline, experiment) f. Call delete_vllm_pod to clean up 3. NEVER kill processes on the baseline pod. NEVER restart the baseline pod. STOPPING CRITERIA: - Keep running until 10 CONSECUTIVE experiments show NO improvement over current best. - Any experiment improving throughput OR latency by >2% resets the counter to zero. - Do NOT stop early.

Tunable Parameters Taught to Claude

VLLM TUNABLE PARAMETERS: 1. --max-num-seqs (1-1024) 2. --max-num-batched-tokens (256-32768) 3. --gpu-memory-utilization (0.80-0.95) 4. --enable-chunked-prefill (bool) 5. --enable-prefix-caching (bool) 6. --max-model-len (int) 7. --enforce-eager (bool) 8. --tensor-parallel-size (1-8) 9. --quantization (fp8/awq/gptq) 10. --scheduling-policy (fcfs/priority)

Mandatory Post-Benchmark Steps

MANDATORY WORKFLOW FOR EACH BENCHMARK: After EVERY run_benchmark call, you MUST: 1. CALL fetch_vllm_logs Parses 120+ regex patterns: server config, engine config, memory, compilation timing, warnings/errors 2. CALL read_benchmark_results Returns per-concurrency metrics: throughput, TTFT, ITL, TPOT (P50/P95/P99)

Engineering Deep Dive

🧪

PodManager

Creates pods from YAML template with modified vLLM args. Generates unique names (vllm-tune-{ts}), polls readiness every 5s, starts oc port-forward on incrementing ports (8001, 8002...). Tracks all procs for cleanup.

🔄

Auto Request Type

GuideLLM needs --request-type. We auto-detect: if model name contains "chat", "instruct", "it-" → chat_completions. Otherwise → text_completions. Fixed 400 errors on base models like opt-125m.

🔍

120+ Log Patterns

Regex parser extracts: vLLM version, non-default args, architecture, dtype, TP size, chunked prefill, attention backend, KV cache size, CUDA graph timing, torch.compile time, model memory, and all warnings/errors.

📊

Directionality-Aware Comparison

compare_benchmarks knows throughput is higher=better, latency is lower=better. 2% threshold. Reports per-metric regressions and improvements across 45 metrics per concurrency level.

🛡

Crash-Safe Cleanup

atexit.register(cleanup_all) + try/finally in main loop. Even if the agent crashes mid-experiment, leftover pods are deleted. Port-forward processes are killed.

💰

Prompt Caching

System prompt uses cache_control: ephemeral. On our 25-iteration run: 294K input tokens, but most were cache hits. Total cost: $0.93 for the full tuning session.

Why So Cheap?

The system prompt (~5K tokens) is marked with cache_control: ephemeral. Across 26 API calls, it's only billed at full price once — subsequent calls use cached input. This gives us 90%+ cache hit rate on the prompt.
